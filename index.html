Wunse CRM · Lovable Build Sequence

24 Prompts.
Build order matters.

Each prompt is self-contained and builds on the previous. Run them in order. Never skip a phase — later prompts reference tables, components, and hooks that earlier prompts establish.

How to use these prompts

  1. One prompt per Lovable session. Don't combine them.
  2. Always connect Supabase before starting. Prompts 02+ assume the connection exists.
  3. After each prompt: test the feature, fix any Lovable errors, then move to the next.
  4. Copy the full prompt text. The specificity is intentional — vague prompts produce generic output.
  5. For styling: the design system is set in Prompt 01. Every subsequent prompt references it.

17
Pipeline velocity calculation engine
SupabaseLogic
▾
Context

Build as a Supabase Edge Function queryable from the frontend. All four inputs calculated from real data. ACV uses ARR only — never mix one-time fees into velocity ACV.

Create a Supabase Edge Function to calculate Pipeline Velocity metrics. EDGE FUNCTION: /functions/v1/pipeline-velocity Accepts: pipeline_type, period_start, period_end, compare_period_start, compare_period_end CALCULATE: 1. QO: COUNT opportunities that entered the QP threshold stage (default 'solution') during the period. Track via stage_history table: (id, opportunity_id, from_stage, to_stage, changed_at, changed_by). Create a trigger: whenever opportunity stage changes, INSERT to stage_history. 2. WR: Won / (Won + Lost) for deals that reached 'proposal' or beyond in period. Return null with insufficient_data flag if denominator < 5. 3. ACV: Average arr_value (ARR only — payment_type IN ('arr','mrr')) from opportunity_products for closed_won opps in period. Return null if < 3 deals. 4. SCL: Average days from created_at to updated_at (when stage=closed_won) for deals closed in period. Per period — NOT all-time average. FORMULA: PV = (QO * WR * ACV) / SCL Returns: { qo, wr, acv, scl, velocity, qo_delta, wr_delta, acv_delta, scl_delta, velocity_delta, insufficient_data_flags }

BCC address format

Every CRM user gets a unique BCC address generated automatically on signup. It identifies which user sent the email.

crm-u-{8-char-uuid}@mail.wunse.com.au

Each contact also gets a unique BCC address for precise attribution:

crm-c-{8-char-uuid}@mail.wunse.com.au

Users find their personal BCC address under Settings → Profile. Save it as a contact in Gmail or Outlook named "Wunse CRM Log" so it autocompletes in one keystroke.

The BCC address only needs to be added once per email thread. Postmark captures all emails sent to that address regardless of which contact they're sent to.

How the Edge Function processes emails
  1. Validates the token in the webhook URL against BCC_WEBHOOK_SECRET in Supabase secrets. Returns 401 if no match. Also accepts a valid Supabase JWT in the Authorization header (used by the CRM test button).
  2. Scans all recipient addresses (To, CC, BCC) for a match against users.bcc_address to identify which CRM user sent the email.
  3. Matches the sender's email address against contacts.email to identify the contact and their company.
  4. Checks for any open lead or opportunity associated with that contact to link the activity.
  5. Creates an activity record — type: email, direction: outbound, subject, body, is_bcc_captured: true, logged_by, contact_id, company_id, opportunity_id if found.
  6. Returns 200 with match status. Returns 200 with "no_match" if no BCC address recognised — Postmark requires a 200 response regardless of outcome.
Monitoring & troubleshooting

The CRM has a built-in capture log at Settings → Email Capture showing the last 50 BCC-captured emails with subject, matched user, matched contact, and timestamp.

In Postmark, the Activity tab on the Default Inbound Stream shows every email received and whether the webhook returned 200 or an error. If emails are failing, check:

  • The webhook URL in Postmark still contains the correct secret token
  • BCC_WEBHOOK_SECRET in Supabase secrets has not been rotated without updating Postmark
  • The Edge Function is deployed — check Supabase → Edge Functions for deployment status
  • The contact's email address exists as a contact record in the CRM — unmatched emails return no_match and are not logged
  • The BCC address domain is mail.wunse.com.au (not mail.wunse.com) — check Settings → Profile
To test end-to-end: send any email to a contact who exists in the CRM, BCC crm-u-xxxxx@mail.wunse.com.au, wait 30 seconds, check the contact's activity timeline.
Onboarding a new team member

When a new user is invited via Settings → Users, their BCC address is automatically generated on signup. They need to:

  • Go to Settings → Profile and copy their BCC address
  • Save it as a contact in Gmail or Outlook — suggest naming it "Wunse CRM Log"
  • BCC it on any client email they want logged in the CRM

No additional DNS or Postmark configuration is required for new users. The existing infrastructure handles all addresses on the mail.wunse.com.au domain automatically.
