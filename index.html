Wunse CRM · Lovable Build Sequence

24 Prompts.
Build order matters.

Each prompt is self-contained and builds on the previous. Run them in order. Never skip a phase — later prompts reference tables, components, and hooks that earlier prompts establish.

How to use these prompts

  1. One prompt per Lovable session. Don't combine them.
  2. Always connect Supabase before starting. Prompts 02+ assume the connection exists.
  3. After each prompt: test the feature, fix any Lovable errors, then move to the next.
  4. Copy the full prompt text. The specificity is intentional — vague prompts produce generic output.
  5. For styling: the design system is set in Prompt 01. Every subsequent prompt references it. Don't re-specify colours in later prompts.

Foundation
Project setup, database schema, authentication, app shell
Start here. This phase produces nothing the user can interact with commercially — it produces the bones. Schema, auth, navigation, and the shared component library that every other prompt builds on. Do not skip or rush this phase.

Pipeline velocity calculation engine
SupabaseLogic
▾
Context

Build this as a Supabase database view or function so it's queryable from the frontend. The four inputs (QO, WR, ACV, SCL) must all be calculated from real data — not hardcoded or manually entered.

ACV uses ARR-only (sum of arr_value from opportunity_products) — never mix one-time fees into the velocity ACV. One-time fees go into TCV only.

Create a Supabase database view and Edge Function to calculate Pipeline Velocity metrics. CREATE DATABASE VIEW: pipeline_velocity_metrics Parameters (passed as query params to the edge function): pipeline_type, period_start, period_end CALCULATE THE FOUR INPUTS: 1. QO (Qualified Opportunities): COUNT of opportunities where: - pipeline_type matches param - The opportunity entered the QP threshold stage (default: 'solution') during the period - Track this by adding a stage_history table: (id, opportunity_id, from_stage, to_stage, changed_at, changed_by) - QO = count of opportunities where a stage_history row shows entry into the QP stage within the period 2. WR (Win Rate): - Numerator: COUNT opportunities closed_won in period where pipeline_type matches - Denominator: COUNT opportunities where (closed_won OR closed_lost) in period AND pipeline_type matches AND stage reached 'proposal' or beyond (to exclude early-stage losses) - WR = numerator / denominator - Return null if denominator < 5 (insufficient data) with a flag "insufficient_data: true" 3. ACV (Average Contract Value — ARR only): - For closed_won opportunities in period: sum of arr_value from opportunity_products / count of those opportunities - ARR only — payment_type IN ('arr', 'mrr'). Exclude one_time, milestone, tm - Return null if < 3 closed won deals (insufficient data) 4. SCL (Sales Cycle Length in days): - For closed_won opportunities in period: average of (updated_at - created_at) in days where stage = 'closed_won' - NOT an all-time average — calculated per period VELOCITY FORMULA: PV = (QO * WR * ACV) / SCL CREATE EDGE FUNCTION: /functions/v1/pipeline-velocity - Accepts: pipeline_type, period_start, period_end, compare_period_start, compare_period_end (for delta calculation) - Returns: { qo, wr, acv, scl, velocity, qo_delta, wr_delta, acv_delta, scl_delta, velocity_delta, insufficient_data_flags } Also create stage_history trigger: whenever an opportunity's stage column changes, INSERT a row into stage_history.
